# 训练数据导出文件清单

## 文件组织结构

```
benchmark_name/
├── static/                          # 静态特征（全 corner 共享，只导一次）
│   ├── graph_enhanced.json          # 增强图结构（edge_index, edge_type）
│   ├── pin_features.json            # 节点静态特征（fanin, fanout, cell_type, pin_role）
│   └── pin_name_mapping.json        # pin_name -> pin_id 映射表（可选，训练时生成）
│
├── anchor_corners/                  # Anchor corners 数据（固定 3 个，用于多锚点融合）
│   ├── ff1p16vn40c/                 # A_fast（⚠️ 也在 train 中）
│   │   ├── arrival.txt              # arrival time (L/R, L/F) - 用于输入特征
│   │   ├── slew.txt                 # slew (L/R, L/F) - 用于输入特征
│   │   ├── pin_cap.txt              # pin capacitance
│   │   ├── net_load.txt             # net load
│   │   └── arc_delay.json           # arc delay（4通道，带mask）- 用于输入特征
│   ├── tt0p85v25c/                  # A_typ（⚠️ 也在 train 中）
│   │   └── ... (同上)
│   └── ss0p7v25c/                   # A_slow（⚠️ 也在 train 中）
│       └── ... (同上)
│
└── target_corners/                  # Target corners 数据（用于监督和评估）
    ├── train/                       # Train corners (15个) ⭐ train_corner 在这里
    │   ├── ff0p85vn40c/
    │   │   ├── arrival.txt          # arrival time (ground truth) - 用于监督信号
    │   │   ├── slack.txt            # slack (评估指标)
    │   │   └── rat.txt              # required arrival time (可选)
    │   ├── ff0p85v25c/
    │   │   └── ... (同上)
    │   ├── ... (共 15 个 train corners)
    │   ├── ff1p16vn40c/             # ⚠️ 也是 Anchor A_fast（重复，但用途不同）
    │   │   ├── arrival.txt          # 作为 target 的 ground truth
    │   │   ├── slack.txt
    │   │   └── rat.txt
    │   ├── tt0p85v25c/              # ⚠️ 也是 Anchor A_typ（重复，但用途不同）
    │   │   └── ... (同上)
    │   └── ss0p7v25c/               # ⚠️ 也是 Anchor A_slow（重复，但用途不同）
    │       └── ... (同上)
    │
    ├── val/                         # Val corners (3个)
    │   ├── ff0p95v25c/
    │   │   └── ... (同上)
    │   ├── ss0p75vn40c/
    │   │   └── ... (同上)
    │   └── tt1p05v25c/
    │       └── ... (同上)
    │
    └── test/                        # Test corners (9个，全部 125°C)
        ├── ff0p85v125c/
        │   └── ... (同上)
        ├── ff0p95v125c/
        │   └── ... (同上)
        └── ... (共 9 个 test corners)
```

**关键说明**：
- **anchor_corners**: 固定 3 个（A_fast, A_typ, A_slow），**用于提供输入特征**
  - 包含：arrival, slew, pin_cap, net_load, arc_delay（完整的输入特征）
  - ⚠️ **这 3 个 corners 也包含在 train corners 中**（见下方）
- **target_corners/train/**: **train_corner 在这里**（15个），**用于训练时的监督信号**
  - 包含：arrival（ground truth）, slack, rat
  - ⚠️ **包含 3 个 anchors**（ff1p16vn40c, tt0p85v25c, ss0p7v25c），但用途不同：
    - 在 `anchor_corners/` 中：作为输入特征源
    - 在 `target_corners/train/` 中：作为监督信号（ground truth）
- **target_corners/val/**: val_corner（3个），用于验证
- **target_corners/test/**: test_corner（9个），用于测试

**数据使用逻辑**：
- 训练时，对于每个 train corner（包括 3 个 anchors）：
  - 输入特征：从 `anchor_corners/` 读取（3 个 anchors 的特征）
  - 监督信号：从 `target_corners/train/` 读取（对应 train corner 的 arrival）

## 详细文件说明

### 1. 静态特征文件（全 corner 共享，只导一次）

#### 1.1 `static/graph_enhanced.json`
**来源**：自定义导出（解析 `dump_graph` + 遍历 arcs）

**内容**：
```json
{
    "nodes": ["U123/A", "U123/Y", "U456/A", ...],
    "edges": [
        {
            "edge_id": 0,
            "src": "U123/Y",
            "dst": "U456/A",
            "type": 0  // 0=cell_arc, 1=net_arc
        },
        ...
    ],
    "edge_consistency_check": {
        "dot_edges_count": 50000,
        "arc_edges_count": 50000,
        "matched_edges": 50000,
        "unmatched_ratio": 0.0  // 必须 < 0.1%
    }
}
```

**⚠️ 关键要求**：
- 必须过滤非 Pin 节点（只保留 Pin 到 Pin 的边）
- 必须进行边一致性检查（DOT 边 vs arcs 边）
- 使用统一的 pin_name 格式（`Inst/Pin`，如 `U123/A`）

#### 1.2 `static/pin_features.json`
**来源**：自定义导出（遍历 pins，获取静态特征）

**内容**：
```json
{
    "U123/A": {
        "fanin": 0,
        "fanout": 1,
        "cell_type": "INV",
        "pin_role": "A"
    },
    "U123/Y": {
        "fanin": 1,
        "fanout": 2,
        "cell_type": "INV",
        "pin_role": "Y"
    },
    ...
}
```

**注意**：`level` 暂不实现

---

### 2. Anchor Corners 动态特征文件（每个 anchor corner 一份）

#### 2.1 `anchor_corners/{corner_name}/arrival.txt`
**来源**：`dump_at [-o arrival.txt]`

**格式**：OpenTimer 默认格式（E/R, E/F, L/R, L/F）

**使用**：提取 **Late 分量**（L/R, L/F），Early 暂不使用

#### 2.2 `anchor_corners/{corner_name}/slew.txt`
**来源**：`dump_slew [-o slew.txt]`

**格式**：OpenTimer 默认格式（E/R, E/F, L/R, L/F）

**使用**：提取 **Late 分量**（L/R, L/F），Early 暂不使用

#### 2.3 `anchor_corners/{corner_name}/pin_cap.txt`
**来源**：`dump_pin_cap [-o pin_cap.txt]`

**格式**：OpenTimer 默认格式（使用 pin_name 作为键）

#### 2.4 `anchor_corners/{corner_name}/net_load.txt`
**来源**：`dump_net_load [-o net_load.txt]`

**格式**：OpenTimer 默认格式

#### 2.5 `anchor_corners/{corner_name}/arc_delay.json` ⚠️ **需要自定义导出**
**来源**：遍历 arcs，访问 `_delay[MAX][rf1][rf2]` 成员

**内容**：
```json
{
    "0": {  // edge_id
        "src": "U123/Y",
        "dst": "U456/A",
        "type": 0,  // 0=cell_arc, 1=net_arc
        "delay_late_rise_rise": 0.3,
        "delay_late_rise_fall": 0.35,
        "delay_late_fall_rise": 0.32,
        "delay_late_fall_fall": 0.37,
        "delay_mask": [1, 1, 1, 1]  // Cell Arc: 所有通道有效
    },
    "1": {
        "src": "U456/Y",
        "dst": "U789/A",
        "type": 1,  // net_arc
        "delay_late_rise_rise": 0.15,  // RR
        "delay_late_rise_fall": 0.0,   // 填充值（无效）
        "delay_late_fall_rise": 0.0,   // 填充值（无效）
        "delay_late_fall_fall": 0.18,  // FF
        "delay_mask": [1, 0, 0, 1]     // Net Arc: 只有 RR 和 FF 有效
    },
    ...
}
```

**⚠️ 关键规则**：
- **Cell Arc**：导出完整 4 通道 `{RR, RF, FR, FF}`，`mask = [1, 1, 1, 1]`
- **Net Arc**：填充到 4 通道，但 `mask = [1, 0, 0, 1]`（只有 RR 和 FF 有效）
- 使用 `pin_name` 作为键（src/dst），训练时转换为 edge_id

---

### 3. Target Corners 标签文件（用于监督和评估）

#### 3.1 `target_corners/{corner_name}/arrival.txt`
**来源**：`dump_at [-o arrival.txt]`（在 target corner 下运行）

**用途**：作为监督信号（ground truth）

**格式**：OpenTimer 默认格式（提取 Late 分量 L/R, L/F）

#### 3.2 `target_corners/{corner_name}/slack.txt`（可选）
**来源**：`dump_slack [-o slack.txt]`

**用途**：评估指标（用于计算 WNS/TNS）

#### 3.3 `target_corners/{corner_name}/rat.txt`（可选）
**来源**：`dump_rat [-o rat.txt]`

**用途**：用于计算 slack（如果需要在训练时计算）

**⚠️ 重要边界声明**：
- **Target corner 的 slack/rat 不作为输入特征**
- **仅用作监督信号（ground truth）和评估指标**
- 避免数据泄露，确保模型真正学习跨 corner 泛化

---

### 4. PVT 编码信息（从库文件名解析）

#### 4.1 `corner_pvt_encoding.json`
**来源**：自定义脚本解析库文件名

**内容**：
```json
{
    "corner_ff_0.70v_25c": {
        "process": "ff",  // one-hot: [1, 0, 0]
        "voltage": 0.70,  // normalize: (0.70 - 0.70) / (1.16 - 0.70) = 0.0
        "temperature": 25  // normalize: (25 - (-40)) / (125 - (-40)) = 0.39
    },
    "corner_ss_0.70v_125c": {
        "process": "ss",  // one-hot: [0, 1, 0]
        "voltage": 0.70,
        "temperature": 125  // normalize: (125 - (-40)) / (125 - (-40)) = 1.0
    },
    ...
}
```

**编码规则**：
- **P (Process)**: one-hot 编码 `ff/ss/tt` → `[1,0,0]` / `[0,1,0]` / `[0,0,1]`
- **T (Temperature)**: 数值归一化 `(-40, 25, 125)` → `[0, 1]`
- **V (Voltage)**: 数值归一化 `(0.70, ..., 1.16)` → `[0, 1]`

## 导出优先级（Phase 实施顺序）

### Phase 1: 基础数据导出（可立即开始）

✅ **必须导出**：
1. `static/graph_enhanced.json`（⚠️ 必须进行边一致性检查）
2. `anchor_corners/{corner}/arrival.txt`（所有 anchor corners）
3. `anchor_corners/{corner}/slew.txt`（所有 anchor corners）
4. `anchor_corners/{corner}/pin_cap.txt`（所有 anchor corners）
5. `anchor_corners/{corner}/net_load.txt`（所有 anchor corners）
6. `corner_pvt_encoding.json`（所有 corners）

### Phase 2: 增强数据导出（需要开发）

⚠️ **第一步：边一致性检查（必须完成）**
- 对 `dump_graph` 的边和遍历 `_arcs` 得到的边进行交集匹配
- 目标：`unmatched_ratio < 0.1%`
- 若 `>1%` 必须定位原因

✅ **然后导出**：
1. `static/pin_features.json`（静态特征，全 corner 共享）
2. `anchor_corners/{corner}/arc_delay.json`（⚠️ 需要自定义导出，使用 mask 机制）

### Phase 3: Target Corner 标签（用于监督）

✅ **导出**：
1. `target_corners/{corner}/arrival.txt`（所有 target corners，作为 ground truth）
2. `target_corners/{corner}/slack.txt`（可选，用于评估）
3. `target_corners/{corner}/rat.txt`（可选，用于计算 slack）

## 数据格式标准化规则

### 1. Pin Name 规范化（必须统一）
- **格式**：统一使用 `Inst/Pin` 格式（如 `U123/A`, `U456/Y`）
- **规则**：去掉引号/转义，保持大小写一致
- **要求**：所有导出文件（DOT、dump_at、自定义导出）必须使用同一套规则

### 2. Edge Features 格式
- ✅ 使用结构化格式：`{"edge_id": 0, "src": "...", "dst": "...", ...}`
- ❌ 避免：使用字符串拼接 `"U123/Y->U456/A"` 作为 key

### 3. Corner 维度说明
- **静态特征**：全 corner 共享（只导一次，与 corner 无关）
- **动态特征**：仅 anchor corners（多份文件，每个 anchor corner 一份）
- **Target 标签**：按 target corner 单独导（用于监督和评估）

## 文件数量统计

假设有：
- 1 个 benchmark
- 3 个 anchor corners（用于输入特征，⚠️ 也包含在 train 中）
- 15 个 train corners（用于监督，包含 3 个 anchors）
- 3 个 val corners（用于验证）
- 9 个 test corners（用于测试）

**总文件数**：
- 静态文件：2 个（`graph_enhanced.json`, `pin_features.json`）
- Anchor corners：3 × 5 = 15 个文件（每个 corner：arrival, slew, pin_cap, net_load, arc_delay）
  - ⚠️ 这 3 个 corners 的数据用于**输入特征**
- Train corners：15 × 2 = 30 个文件（每个 corner：arrival, slack）
  - ⚠️ 包含 3 个 anchors，但数据用于**监督信号**（ground truth）
- Val corners：3 × 2 = 6 个文件（每个 corner：arrival, slack）
- Test corners：9 × 2 = 18 个文件（每个 corner：arrival, slack）
- 配置：1 个（`corner_pvt_encoding.json`）

**总计**：约 72 个文件（每个 benchmark）

**重要说明**：
- 3 个 anchors（ff1p16vn40c, tt0p85v25c, ss0p7v25c）**同时出现在两个地方**：
  1. `anchor_corners/`：导出完整的输入特征（arrival, slew, arc_delay 等）
  2. `target_corners/train/`：导出监督信号（arrival, slack 作为 ground truth）
- **用途不同**：
  - `anchor_corners/` 的数据：用于模型输入（多锚点融合）
  - `target_corners/train/` 的数据：用于训练监督（loss 计算）
