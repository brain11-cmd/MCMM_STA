# 模型输入需求与 OpenTimer 导出能力对比

## 模型输入需求

### 1. 图结构 (Graph Structure)

**需求：**
- **节点 (Nodes)**: pin
- **边 (Edges)**: timing arc (cell arc 或 net arc)
- **Edge Index**: `src_pin_name, dst_pin_name, edge_type`

**OpenTimer 支持：**
- ✅ `dump_graph [-o <file>]` - 导出 DOT 格式的图结构
  - 包含所有 pin 节点（使用 `pin._name`，如 `U123/A`, `U456/Y`）
  - 包含所有 arc 边（`from -> to`，使用 pin_name）
  - ⚠️ 但需要解析 DOT 格式，且没有直接提供 `edge_type`

**需要补充：**
- 需要自定义导出脚本，输出：
  - `edge_index`: `[src_pin_name, dst_pin_name, edge_type]`
  - `edge_type`: 0=cell_arc, 1=net_arc (通过 `Arc::is_cell_arc()` 和 `Arc::is_net_arc()` 判断)
- **训练时**: 使用 `pin_name -> pin_id` 映射字典，一次性转换为整数 ID

**⚠️ 关键注意事项：**

1. **过滤非 Pin 节点**：
   - `dump_graph` 的 DOT 输出可能包含内部工具节点或分组节点（取决于 OpenTimer 版本/配置）
   - **必须添加过滤规则**：只保留 Pin 到 Pin 的边（两端都能在 `pin_name` 集合中匹配）
   - 忽略非 pin 节点，避免引入"虚拟节点"到图结构中

2. **边的一致性检查（Phase 2 第一步必须做）**

**权威数据源**：
- ✅ **以遍历 `_arcs` 导出的边为权威**
- ⚠️ **DOT 仅用作辅助可视化/交叉校验**

**原因**：避免后续发现 DOT 不一致时陷入"到底信哪个"的争论。

**一致性检查流程**：
1. 遍历 `_arcs` 得到权威边集合（通过 `src_pin_name`, `dst_pin_name`）
2. 解析 `dump_graph` 的 DOT 输出得到参考边集合
3. 进行交集匹配，统计未匹配边的比例
4. **两级阈值**：
   - **硬阈值**：`unmatched_ratio < 1%`（超过就一定有问题，必须定位原因）
   - **目标阈值**：`< 0.1%`（跑通后再精修，第一次跑通时 DOT 解析、命名规范化、clock pin 处理可能 >0.1%）
5. 如果 DOT 不一致，以 `_arcs` 为准，DOT 仅用于警告提示

**自动化校验（必须执行）**：

**覆盖率检查**：
- 检查：`dump_at` 的 pin 集合 vs 导出的 `node_static` pin 集合
- 要求：`|intersection| / |dump_at| > 99.9%`（或至少 `>99%` 先跑通）
- 目的：确保导出的节点覆盖了所有有时序意义的 pin

**唯一性检查**：
- 检查：规范化后 `pin_name` 必须唯一
- 要求：如果出现冲突，立刻打印冲突样本（原始字符串列表）
- 目的：避免 mapping 崩溃（`pin_name -> pin_id` 映射必须是一对一的）

### 2. 节点特征 (Node Features)

#### 2.1 静态结构特征 (Static Structural Features) - 与 corner 无关

**需求：**
- `fanin` - 输入边数量
- `fanout` - 输出边数量
- `cell_type` (embedding) - 单元类型
- `pin_role` - 引脚角色 (如 A/Y/CK/Q)

**OpenTimer 支持：**
- ✅ `Pin::num_fanins()` - 可以获取 fanin 数量
- ✅ `Pin::num_fanouts()` - 可以获取 fanout 数量
- ✅ `Pin::cellpin(Split)` - 可以获取 Cellpin，进而获取：
  - `Cellpin::name()` - pin_role (如 "A", "Y", "CK", "Q")
  - `Cellpin::cell()` - 可以获取 Cell，进而获取 `Cell::name()` - cell_type

**需要补充：**
- 需要自定义导出脚本，导出：
  - `fanin`, `fanout` (直接可用)
  - `cell_type` (从 Cellpin 获取)
  - `pin_role` (从 Cellpin 获取)
- **注意**: `level` 暂不实现，后续如需要再添加

#### 2.2 动态 STA 状态变量 (Dynamic STA State) - 仅从 anchor corners

**需求：**
- `arrival` - 到达时间
- `slew` - 转换时间

**OpenTimer 支持：**
- ✅ `dump_at [-o <file>]` - 导出 arrival time
  - 格式：`E/R, E/F, L/R, L/F` (Early/Late, Rise/Fall)
- ✅ `dump_slew [-o <file>]` - 导出 slew
  - 格式：`E/R, E/F, L/R, L/F`

**默认使用策略（Phase 1）：**
- **使用 Late 分量**（L/R, L/F），Early 暂不使用
- **Rise/Fall 双通道**：分别使用 `late_rise` 和 `late_fall`，**不使用 max(R, F)**
- **理由**：
  - Late 对应 setup time，是主要约束
  - Rise/Fall 差异显著，GNN 可以学习这种差异
  - 后续可扩展到 Early 或完整 4 通道

**数据格式：**
```python
node_features = {
    "arrival_late_rise": float,   # L/R
    "arrival_late_fall": float,   # L/F
    "slew_late_rise": float,      # L/R
    "slew_late_fall": float,      # L/F
}
```

**⚠️ 重要边界声明：**
- **Target corner 的 slack/rat 不作为输入特征**
- **仅用作监督信号（ground truth）和评估指标**
- 避免数据泄露，确保模型真正学习跨 corner 泛化

**可以直接使用：**
- 从 anchor corners 运行 OpenTimer，使用 `dump_at` 和 `dump_slew` 导出

### 3. 边特征 (Edge Features)

#### 3.1 Arc-level 特征 (从 Anchor Corners)

**需求：**
- `arc_delay` - 弧延迟
  - 区分 cell/net
  - 可以包含 rise/fall, min/max

**OpenTimer 支持：**
- ✅ Arc 内部有 `_delay[el][rf1][rf2]` 存储延迟
  - `el`: MIN/MAX (early/late)
  - `rf1`: RISE/FALL (from pin transition)
  - `rf2`: RISE/FALL (to pin transition)
- ❌ **没有直接的 `dump_arc_delay` 命令**

**Arc Delay 通道定义（固定规则）：**

**⚠️ 关键发现（基于 OpenTimer 源码分析）：**

1. **Cell Arc（完整的 4 通道，Late/MAX）：**
   - ✅ **Cell Arc 在 Late/MAX 下具有完整 4 通道延迟**：`{RR, RF, FR, FF}`
   - 源码：`_delay[el][frf][trf] = tv[el]->delay(frf, trf, si, lc)`
   - 格式：`arc_delay_late_rise_rise`, `arc_delay_late_rise_fall`, `arc_delay_late_fall_rise`, `arc_delay_late_fall_fall`
   - 对应：`_delay[MAX][RISE][RISE]`, `_delay[MAX][RISE][FALL]`, `_delay[MAX][FALL][RISE]`, `_delay[MAX][FALL][FALL]`
   - **原因**：Cell 内部逻辑可能改变 transition 方向（如 INV）

2. **Net Arc（仅同向传播通道，Late/MAX）：**
   - ✅ **Net Arc 的延迟仅定义同向传播通道**：`RR` 与 `FF`
   - ❌ **RF/FR 在 OpenTimer 中为 `std::nullopt`**
   - 源码：`_delay[el][rf][rf] = net->_delay(el, rf, _to)`（注意：只有 `[rf][rf]`）
   - **原因**：Net 只是传输信号，**不会改变 transition 方向**

**本工作的通道使用策略（Late/MAX 设置）：**

✅ **统一导出 4 通道 + mask**：
- **Cell Arc**：4 通道 `{RR, RF, FR, FF}`，`mask = [1, 1, 1, 1]`（所有通道有效）
- **Net Arc**：填充到 4 通道 `{RR, RF, FR, FF}`，但 `mask = [1, 0, 0, 1]`（只有 RR 和 FF 有效）
- **RF/FR 使用 mask 置为无效**：`mask[RF] = 0`, `mask[FR] = 0`
- **在 loss/传播中跳过无效通道**：计算 loss 和 forward 传播时，忽略 mask=0 的通道

**Max-Plus 传播规则：**

✅ **Max-plus 传播对 Net Arc 强制同向更新**：
- 仅允许 `R→R` 与 `F→F`，不允许跨跃迁方向
- 源码验证：`_fprop_at()` 中 `FOR_EACH_EL_RF_RF_IF(el, frf, trf, _from._at[el][frf] && _delay[el][frf][trf])`
- 对于 Net Arc，只有 `_delay[el][rf][rf]` 有值，因此只传播 `rf→rf`

**优点**：
- ✅ 统一维度，便于 GNN 实现（无需条件分支）
- ✅ Mask 机制常见，PyTorch Geometric 等框架支持
- ✅ 保持数据真实性（mask 明确标记无效通道）

**需要补充：**
- 需要自定义导出脚本或修改 OpenTimer 代码
- 遍历所有 arcs，导出：
  - `src_pin_name`, `dst_pin_name` (使用 pin_name 作为键)
  - `edge_type` (cell_arc=0, net_arc=1)
  - **Cell Arc**: `arc_delay_late_rise_rise`, `arc_delay_late_rise_fall`, `arc_delay_late_fall_rise`, `arc_delay_late_fall_fall`
  - **Net Arc**: 
    - **默认**：RR = `_delay[MAX][RISE][RISE]`, FF = `_delay[MAX][FALL][FALL]`（OpenTimer 已提供）
    - RF/FR：填 0 + `mask=0`
    - **仅当 RR/FF 也缺失时**（极少数数据缺失）：才使用 `dNet_max` 近似

#### 3.2 可选物理特征 (Optional Physical Features)

**需求：**
- `RC/load` - 如果可用；否则通过拓扑和 anchor delay/slew 近似

**OpenTimer 支持：**
- ✅ `dump_pin_cap [-o <file>]` - 导出 pin capacitance
- ✅ `dump_net_load [-o <file>]` - 导出 net load
- ✅ `dump_rctree` - 导出 RC tree（如果有 SPEF 文件）

**可以直接使用：**
- 从 anchor corners 运行 OpenTimer，使用上述命令导出

#### 3.3 Net Arc 退化策略

**处理策略（按优先级）：**
- **默认**：RR/FF + mask（RF/FR 无效，mask=[1,0,0,1]）
- **缺失时**：使用 `dNet_max` 作为补缺（仍用 mask）
- **最后**：才考虑拓扑近似（可选）

### 4. 目标 Corner 条件 (Target Corner Conditions) - PVT 编码

**需求：**
- `z_c = [P, V, T]` 编码 (one-hot/embedding + normalization)
  - **P (Process)**: one-hot 编码 `ff/ss/tt`
  - **T (Temperature)**: 数值归一化 `(-40/25/125)`
  - **V (Voltage)**: 数值归一化 `(0.70/0.75/.../1.16)`
  - 注意：不需要强制要求 ff/ss/tt 的 V 档一致，直接当连续变量

**OpenTimer 支持：**
- ✅ 可以从库文件名解析 PVT 信息

**需要补充：**
- 需要自定义脚本解析库文件名，提取 PVT 信息
- 实现编码逻辑：
  ```python
  # P: one-hot
  P_ff = [1, 0, 0]
  P_ss = [0, 1, 0]
  P_tt = [0, 0, 1]
  
  # T: normalize (-40, 25, 125)
  T_norm = (T - (-40)) / (125 - (-40))  # 归一化到 [0, 1]
  
  # V: normalize (0.70, ..., 1.16)
  V_norm = (V - 0.70) / (1.16 - 0.70)  # 归一化到 [0, 1]
  ```

## OpenTimer 现有导出命令总结

### 可直接使用的命令：

1. **图结构**:
   - `dump_graph [-o <file>]` - DOT 格式（需要解析）

2. **节点特征**:
   - `dump_at [-o <file>]` - arrival time (E/R, E/F, L/R, L/F)
   - `dump_slew [-o <file>]` - slew (E/R, E/F, L/R, L/F)
   - `dump_slack [-o <file>]` - slack (E/R, E/F, L/R, L/F)
   - `dump_rat [-o <file>]` - required arrival time (E/R, E/F, L/R, L/F)
   - `dump_pin_cap [-o <file>]` - pin capacitance

3. **边/网络特征**:
   - `dump_net_load [-o <file>]` - net load
   - `dump_rctree` - RC tree（如果有 SPEF）

### 需要自定义导出的内容：

1. **图结构增强**:
   - `edge_index`: `[src_pin_name, dst_pin_name, edge_type]` (使用 pin_name)
   - 训练时建立 `pin_name -> pin_id` 映射字典

2. **节点静态特征**:
   - `fanin`, `fanout` (可通过 API 获取)
   - `cell_type` (需要从 Cellpin 提取)
   - `pin_role` (需要从 Cellpin 提取)
   - **注意**: `level` 暂不实现

3. **边特征**:
   - `arc_delay` (需要遍历 Arc 并访问 `_delay` 成员，使用 pin_name 作为键)

4. **PVT 编码**:
   - 从库文件名解析并编码

## 建议的数据导出方案

### 方案 1: 修改 OpenTimer 添加导出命令（推荐）

在 OpenTimer 中添加新的导出命令：
- `dump_graph_enhanced [-o <file>]` - 导出增强的图结构（包含 edge_index, edge_type，使用 pin_name）
- `dump_pin_features [-o <file>]` - 导出节点静态特征（fanin, fanout, cell_type, pin_role，使用 pin_name）
- `dump_arc_delay [-o <file>]` - 导出 arc delay（使用 pin_name 作为键）

### 方案 2: 使用 OpenTimer API 编写自定义导出脚本

使用 OpenTimer 的 C++ API 或通过 Python 绑定（如果有）编写脚本：
- 读取 OpenTimer 的内部数据结构
- 遍历 pins 和 arcs
- 导出所需的所有特征

### 方案 3: 解析现有 dump 输出 + 补充计算

- 解析 `dump_graph` 输出获取图结构（使用 pin_name）
  - **⚠️ 必须过滤**：只保留 Pin 到 Pin 的边（两端都在 pin_name 集合中）
  - **⚠️ 必须验证**：确保 DOT 边与遍历 arcs 得到的边一致
- 解析 `dump_at`, `dump_slew` 等获取动态特征（使用 pin_name 作为键）
- 从 Cellpin 提取 `cell_type`, `pin_role` 等
- 通过其他方式获取 `arc_delay`（可能需要修改 OpenTimer，使用 pin_name 作为键）

## 实施优先级

### Phase 1: 基础数据导出（可立即开始）
1. ✅ 使用现有命令导出：
   - `dump_at`, `dump_slew` (从 anchor corners)
   - `dump_pin_cap`, `dump_net_load`
2. ⚠️ 解析 `dump_graph` 获取图结构
   - **必须过滤非 Pin 节点**：只保留 Pin 到 Pin 的边
   - **必须验证边的一致性**：确保 DOT 边与 arcs 一致
3. ⚠️ 实现 PVT 编码脚本

### Phase 2: 增强数据导出（需要开发）
1. **⚠️ 第一步：边的一致性检查（必须完成）**
   - 对 `dump_graph` 的边和遍历 `_arcs` 得到的边进行交集匹配
   - 通过 `src_pin_name`, `dst_pin_name` 进行匹配
   - 统计未匹配边的比例，确保对齐
   - **目标**：`unmatched_ratio < 0.1%`（越低越好）
   - **阈值**：若 `>1%` 必须定位原因（可能是导出逻辑错误或数据不一致）
   - **这是最关键的一步，必须在其他导出之前完成**

2. **Pin Name 规范化（必须统一）**
   - 统一使用 `Inst/Pin` 格式（如 `U123/A`）
   - 去掉引号/转义，保持大小写一致
   - 确保 DOT、dump_at、自定义导出三者使用同一套规则
   - 建立 pin_name 规范化函数，所有导出统一调用

3. 导出节点静态特征（fanin, fanout, cell_type, pin_role，使用 pin_name）
   - **注意**：静态特征全 corner 共享，只导一次

4. 导出 arc_delay（使用 pin_name 作为键）
   - **Cell Arc**：导出 4 通道 {RR, RF, FR, FF}（Late/Max）
   - **Net Arc**：若无法获得 4 组合，导出 dNet_max 并广播为 4 通道
   - **或**：先只对 cell_arc 做预测，net_arc 直接使用 OpenTimer 的 net delay

### Phase 3: 数据格式统一
1. 统一数据格式（如 JSON, HDF5, PyTorch Geometric 格式）
2. 数据验证和完整性检查
3. **Corner 维度说明**：
   - **静态特征**：全 corner 共享（只导一次，与 corner 无关）
   - **动态特征**：仅 anchor corners（多份文件，每个 anchor corner 一份）
   - **Target 标签**：按 target corner 单独导（用于监督和评估）

### 推荐的文件格式（最推荐的最小集合 - CSV格式）

**目标**：让 PyTorch 端最省事，每个 benchmark 一套文件

**文件列表**：

1. **`meta.json`** - 元数据（必须）
   ```json
   {
       "time_unit": "ns",
       "benchmark_name": "tinyRocket",
       "num_nodes": 50000,
       "num_edges": 60000,
       "missing_structural": 0.25,
       "missing_unexpected": 0.001
   }
   ```
   - `missing_structural`: Net Arc 的 RF/FR 结构性缺失率（正常，预计占比很大）
   - `missing_unexpected`: 其他意外缺失率（需报警，不应 >5%）

2. **`graph_edges.csv`** - 图结构（静态，全 corner 共享）
   ```csv
   edge_id,src,src_id,dst,dst_id,edge_type
   0,U123/Y,100,U456/A,200,0
   1,U456/Y,201,U789/A,300,1
   ...
   ```
   - `edge_id`: 边 ID（从 0 开始连续编号）
   - `src`, `dst`: 源/目标 pin_name（字符串，用于调试）
   - `src_id`, `dst_id`: 源/目标 node_id（整数，可选但推荐，训练更省事）
   - `edge_type`: 0=cell_arc, 1=net_arc
   - **注意**：`src_id`/`dst_id` 对应 `node_static.csv` 中的 `node_id`

3. **`node_static.csv`** - 节点静态特征（静态，全 corner 共享）
   ```csv
   node_id,pin_name,fanin,fanout,cell_type_id,pin_role_id
   0,U123/A,0,1,5,0
   1,U123/Y,1,2,5,1
   ...
   ```
   - `node_id`: 节点 ID（从 0 开始连续编号，**按 pin_name 排序后生成，稳定可复现**）
   - `pin_name`: pin 名称（字符串）
   - `cell_type_id`: cell_type 的整数编码（如 INV=5, NAND2=10）
   - `pin_role_id`: pin_role 的整数编码（如 A=0, Y=1, CK=2）
   - **排序规则**：按 `pin_name` 字典序排序后编号，确保跨 corner 一致
   - **优点**：训练端可直接 mmap/读取，不用每次再做字符串查表

4. **`anchor_node_dyn_{k}.csv`** - Anchor 节点动态特征（k=0,1,2，对应 3 个 anchors）
   ```csv
   pin_name,at_LR,at_LF,slew_LR,slew_LF
   U123/A,0.5,0.6,0.1,0.12
   ...
   ```
   - `at_LR`: arrival time Late Rise
   - `at_LF`: arrival time Late Fall
   - `slew_LR`: slew Late Rise
   - `slew_LF`: slew Late Fall
   - **单位**：ns（在 meta.json 中声明）

5. **`anchor_edge_delay_{k}.csv`** - Anchor 边延迟（k=0,1,2，对应 3 个 anchors）
   ```csv
   edge_id,dRR,dRF,dFR,dFF,maskRR,maskRF,maskFR,maskFF
   0,0.3,0.35,0.32,0.37,1,1,1,1
   1,0.15,0.0,0.0,0.18,1,0,0,1
   ...
   ```
   - `dRR`, `dRF`, `dFR`, `dFF`: delay 的 4 通道
   - `maskRR`, `maskRF`, `maskFR`, `maskFF`: 对应通道的 mask（1=有效，0=无效）
   - **缺失值处理**：缺失时填 0，对应 mask=0
   - **单位**：ns（在 meta.json 中声明）

6. **`target_edge_delay.csv`** - Target 边延迟（用于监督）
   ```csv
   edge_id,corner_name,dRR,dRF,dFR,dFF,maskRR,maskRF,maskFR,maskFF
   0,ff0p85v25c,0.31,0.36,0.33,0.38,1,1,1,1
   0,ff0p95v25c,0.32,0.37,0.34,0.39,1,1,1,1
   ...
   ```
   - 格式同 `anchor_edge_delay_{k}.csv`，但包含 `corner_name` 列
   - 用于训练时的监督信号（ground truth）

7. **`target_node_arrival.csv`** - Target 节点 arrival（可选，Phase 2）
   ```csv
   pin_name,corner_name,at_LR,at_LF,slew_LR,slew_LF
   U123/A,ff0p85v25c,0.51,0.61,0.11,0.13
   ...
   ```
   - 用于 arrival 监督（可选，Phase 2 使用）

**优点**：
- ✅ CSV 格式简单，PyTorch 可直接用 `pandas.read_csv()`
- ✅ 文件分离清晰，便于按需加载
- ✅ 元数据集中管理（单位、缺失率等）
- ✅ 最小集合，避免冗余

**注意事项**：
- 所有时间相关字段（delay/arrival/slew）必须统一单位（ns）
- 缺失值统一处理：填 0 + mask=0
- 缺失率需分级记录在 `meta.json` 中（`missing_structural` 和 `missing_unexpected`）
- `node_id` 按 `pin_name` 排序后生成，确保稳定可复现
- `graph_edges.csv` 中可选的 `src_id`/`dst_id` 便于训练时直接使用，无需字符串查表

### Phase 4: 传播与 Slack 计算（可选增强）

**Arrival 传播：**
- Arrival 由 max-plus 算法得到（STA 标准传播）
- 使用预测的 arc_delay 在图上做一次 STA 传播

**Slack 计算：**
- Slack 需要 RAT（Required Arrival Time）
- RAT 可通过 `dump_rat` 导出，或从时钟约束计算
- **建议**：Phase 2 中 arrival/slack 都做监督（更稳定）

**监督信号：**
- **Target corner 的 arrival**：作为监督信号（ground truth）
- **Target corner 的 slack**：作为监督信号和评估指标
- **注意**：slack/rat 不作为输入特征，避免数据泄露

**⚠️ 数据格式标准化规则：**

1. **单位统一（必须）**
   - **要求**：所有 delay/arrival/slew 使用同一单位（建议 ns）
   - **实现**：在 `meta.json` 中写入 `time_unit = "ns"`，并在导出脚本中固定转换
   - **自检**：不同 corner 同一条 arc 的 delay 数量级应合理（不会有的 0.0001 有的 100）
   - **重要性**：否则 loss 的 eps、log1p、相对误差会混乱

2. **缺失值处理（必须固定策略）**
   - **问题**：OpenTimer 的 `_delay`、某些 pin 的 arrival/slew 可能出现缺失（`std::nullopt`）
   - **策略**：任何缺失值导出为 `0` + `mask=0`（或额外 `valid` 字段）
   - **缺失原因分级**（必须区分）：
     - **结构性缺失（正常）**：Net Arc 的 RF/FR 本来就无定义（`missing_structural`）
     - **意外缺失（需报警）**：
       - Cell Arc 的四通道缺失（RR/RF/FR/FF 任一缺失）
       - Net Arc 的 RR/FF 缺失
       - Pin 的 arrival/slew 缺失
       - 可能原因：未 `update_timing`、约束不完整、pin 不可达/无时序意义
   - **统计**：分别记录 `missing_structural` 和 `missing_unexpected` 写入 `meta.json`
   - **自检**：`missing_unexpected` 不应太高（如 `>5%` 需查原因）

3. **Edge Features 不使用字符串拼接作为 key**
   - ❌ 避免：`"U123/Y->U456/A"` 作为 key
   - ✅ 使用：结构化格式 `{"edge_id": 0, "src": "...", "dst": "...", ...}`
   - 或使用 `edge_id` 作为 key，在 edges 中存储 src/dst 映射

4. **Pin Name 规范化规则（必须统一）**
   - **格式**：统一使用 `Inst/Pin` 格式（如 `U123/A`, `U456/Y`）
   - **去掉引号/转义**：DOT 解析时去除引号
   - **保持大小写一致**：所有导出（DOT、dump_at、自定义导出）必须使用同一套规则
   - **验证**：确保 DOT、dump_at、自定义导出三者的 pin_name 完全一致

        nonlocal missing_structural, missing_unexpected

            if is_structural:
                missing_structural += 1
            else:

            return default, 0  # 返回默认值和 mask=0

        dFR, maskFR = get_delay_value(arc._delay[MAX][FALL][RISE], is_structural=False)
        dFF, maskFF = get_delay_value(arc._delay[MAX][FALL][FALL], is_structural=False)

            "delay_late_rise_fall": dRF,
            "delay_late_fall_rise": dFR,

## 关键设计决策

### 使用 Pin Name 而非 Pin ID

**决定**: 使用 `pin_name`（字符串）作为数据导出的主键，训练时转换为整数 ID。

**理由**:
1. ✅ **无需修改 OpenTimer**: `dump_graph` 等命令已输出 pin_name
2. ✅ **跨 corner 对齐简单**: 同一 benchmark 在不同 corner 下 pin_name 相同
3. ✅ **调试方便**: 数据文件可读性强
4. ✅ **性能影响可忽略**: 训练时一次性映射，开销很小

**实现**:
- 导出阶段: 使用 pin_name（如 `U123/A`, `U456/Y`）
- 训练阶段: 建立 `pin_name -> pin_id` 映射字典，一次性转换

### Level 暂不实现

**决定**: `level`（逻辑层级）暂不实现，后续如需要再添加。

**理由**:
- 需要拓扑排序计算，增加实现复杂度
- 当前阶段优先保证核心功能
- 如模型需要，后续可以补充


## 关键实现规则总结

### 1. 时序特征使用策略
- **Phase 1 默认**：使用 Late 分量（L/R, L/F），Early 暂不使用
- **Rise/Fall**：双通道分别使用，不使用 max(R, F)
- **Target corner 的 slack/rat**：不作为输入，仅用作监督/评估

### 2. Arc Delay 通道定义（统一方案）
- **Cell Arc**：在 Late/MAX 下具有完整 4 通道延迟 {RR, RF, FR, FF}
- **Net Arc**：延迟仅定义同向传播通道 {RR, FF}；RF/FR 在 OpenTimer 中为 `std::nullopt`
- **统一策略**：使用 mask 机制统一到 4 通道
  - Cell Arc：4 通道，`mask = [1, 1, 1, 1]`（所有通道有效）
  - Net Arc：填充到 4 通道，`mask = [1, 0, 0, 1]`（只有 RR 和 FF 有效）
- **Max-Plus 传播**：对 Net Arc 强制同向更新，仅允许 R→R 与 F→F，不允许跨跃迁方向
- **Loss/传播**：计算 loss 和 forward 传播时，忽略 mask=0 的通道

### 3. 边导出权威性（必须明确）
- **权威数据源**：以遍历 `_arcs` 导出的边为权威
- **DOT 用途**：仅用作辅助可视化/交叉校验
- **原因**：避免后续发现 DOT 不一致时陷入"到底信哪个"的争论
- **一致性检查**：DOT 与 `_arcs` 不一致时，以 `_arcs` 为准，DOT 仅用于警告提示

### 4. 自动化校验（必须执行）
- **覆盖率检查**：`dump_at` 的 pin 集合 vs `node_static` pin 集合
  - 要求：`|intersection| / |dump_at| > 99.9%`（或至少 `>99%` 先跑通）
  - 目的：确保导出的节点覆盖了所有有时序意义的 pin
- **唯一性检查**：规范化后 `pin_name` 必须唯一
  - 要求：如果出现冲突，立刻打印冲突样本（原始字符串列表）
  - 目的：避免 mapping 崩溃（`pin_name -> pin_id` 映射必须是一对一的）

### 5. Pin Name 规范化规则
- **格式**：统一使用 `Inst/Pin` 格式（如 `U123/A`）
- **规则**：去掉引号/转义，保持大小写一致
- **要求**：DOT、dump_at、自定义导出三者必须使用同一套规则
- **node_id 生成**：按 `pin_name` 字典序排序后编号（稳定、可复现）

### 5. 单位统一（必须）
- **要求**：所有 delay/arrival/slew 使用同一单位（建议 ns）
- **实现**：在 `meta.json` 中写入 `time_unit = "ns"`，并在导出脚本中固定转换
- **自检**：不同 corner 同一条 arc 的 delay 数量级应合理（不会有的 0.0001 有的 100）
- **重要性**：否则 loss 的 eps、log1p、相对误差会混乱

### 6. 缺失值处理（必须固定策略 + 分级统计）
- **问题**：OpenTimer 的 `_delay`、某些 pin 的 arrival/slew 可能出现缺失（`std::nullopt`）
- **策略**：任何缺失值导出为 `0` + `mask=0`（或额外 `valid` 字段）
- **缺失原因分级**（必须区分）：
  - **结构性缺失（正常）**：Net Arc 的 RF/FR 本来就无定义（`missing_structural`）
  - **意外缺失（需报警）**：
    - Cell Arc 的四通道缺失（RR/RF/FR/FF 任一缺失）
    - Net Arc 的 RR/FF 缺失
    - Pin 的 arrival/slew 缺失
    - 可能原因：未 `update_timing`、约束不完整、pin 不可达/无时序意义
- **统计**：分别记录 `missing_structural` 和 `missing_unexpected` 写入 `meta.json`
- **自检**：`missing_unexpected` 不应太高（如 `>5%` 需查原因）

### 7. 数据格式标准化 + Node ID
- **Edge Features**：使用结构化格式 `{"edge_id": 0, "src": "...", "dst": "...", ...}`
- **避免**：使用字符串拼接 `"U123/Y->U456/A"` 作为 key
- **推荐格式**：CSV 格式（见"推荐的文件格式"章节），便于 PyTorch 加载
- **Node ID**：
  - `node_static.csv` 必须包含 `node_id` 列（按 `pin_name` 排序后编号）
  - `graph_edges.csv` 可选包含 `src_id`/`dst_id`（便于训练时直接使用，无需字符串查表）
  - **排序规则**：按 `pin_name` 字典序排序后编号，确保稳定可复现

### 8. Corner 维度说明
- **静态特征**：全 corner 共享（只导一次）
- **动态特征**：仅 anchor corners（多份文件）
- **Target 标签**：按 target corner 单独导

### 9. Arc 表示统一策略
- **关键发现**：Net Arc 延迟仅定义同向传播通道（RR, FF）；RF/FR 为 `std::nullopt`
- **统一方案**：使用 mask 机制统一 Cell Arc 和 Net Arc 到 4 通道
  - Cell Arc：4 通道，`mask = [1, 1, 1, 1]`
  - Net Arc：填充到 4 通道，`mask = [1, 0, 0, 1]`
- **Max-Plus 传播**：对 Net Arc 强制同向更新，仅允许 R→R 与 F→F
- **Loss/传播**：计算 loss 和 forward 传播时，忽略 mask=0 的通道
- **优点**：统一维度，便于 GNN 实现；保持数据真实性（mask 明确标记无效通道）
